    Step 1: Backfill for each max-flow pathing the amount of enemy land that can be filled for different turn/army combos. This step will produce a lookup table that we will use to finalize a good combination of flows.
        It should produce a class called a "FlowArmyTurnsLookupTable" which should contain all the data covered below after this step 1 algorithm runs. Lets call this algorithm "process_flow_into_flow_army_turns(data: FlowGraphData,...) -> FlowArmyTurnsLookupTable"
        EG For the friendly-target border islands a1 and b1, b1's best tiles/army capture options are 7 turns of capturing 1s, 2 turns of capturing 2s, a 7, and then 6 more 1's or something.
            Knapsack-order this so we can always just see a lookup table of 18 army at this border gets us 12 captures, 25 army at this border gets us 15 captures etc.
                For each border build a lookup table by tile count (just raw array, indexed by turn number since we never search more than 50 turns generally).
                Lookup table returns the amount of army needed for that many capture turns (the lookup'd turn), and the economic value of those captures as advized by the capture value matrix, AND ALSO the combination of island / flow nodes that achieved that amount, plus the capture value matrix.
            These are always unique sets, borders tile counts cannot include any tiles that are included by another border.
        We also track the amount of gatherable up the flow chain from the friendly side, too, doing the exact same thing for the gather side.
            We crawl each side with a visited set per-border. We crawl the highest value per tile STREAM friendly nodes first (referencing the flow numbers we calculated in the max-flow streams), and the lowest (isNeutral, armyPerTile) target nodes first (so prefer enemy islands first always, picking low-army-tiles first, falling back to neutral).
                Step 1.5: "preprocess_flow_stream_tilecounts(data: FlowGraphData, ...)" which will compute additional data into FlowGraphData. This step can really be combined to happen right after the max-flow step results are processed back into FlowGraphData.
                    For calculating the highest value per tile streams, we will need to count the friendly tile count in a given stream, and then divide the army flow at the border by the tile count. As the stream divides upward (since many streams join together on the friendly side), we need to continue selecting the highest flow/tile count up the stream. So each flow node needs to track its upstream tile counts and upstream flow amounts.
                        We may add some pre-processing on this later to pre-prune bad flow-starts, like patches of 2's from our land that merge with high value streams, in order to improve the numbers for the other parts of the streams.
            SO following ^ we crawl upwards preferring to check the highest value streams first, and tracking how many tiles we've "included" for sure in each stream (so guaranteed the border friendly island tiles, then as we walk upwards in the search we include that nodes tile count etc).
        Then if we have 10 border pairs like this, this guarantees we can save 10 (20..?) moves joining and splitting by flowing through as many borders as possible compared to gathering all our land and flowing through one border crossing and fanning back out to capture.
            (since each border we flow army through saves two(? at least one) moves to capture the tiles it flows to (vs gathering from this flow, joining another flow,
            crossing another flow border, then crossing back from the other enemy flow into this enemy flow - two moves lost. One if you dont gather any from this flow and just capture between the flows)).
        With the visited set preventing us from re-visiting nodes that we already visited this should be a very quick algorithm to run (which should be able to be disabled in order to brute force the best solutions should we need to cross reference a bad output against the true optimal solution for a graph).
        Note that this is not guaranteed to find the optimal solution as the highest flow/value stream is not necessarily the highest value stream for the number of turns we may end up needing to use from this stream, so we may visit a suboptimal order in specific edge cases (which non-visited brute force should be able to root out).

            Beginning at each border pair, by walking down the gather and capture tables for that border crossing in parallel, find the maximum of (the econ value of the capture table entry / (number of tiles in the capture table entry + number tiles gathered by the army entry in the gather table that exceeds the capture army requirement for the current walked-to capture army entry)).

    Step 3: We will have a flag that simply returns the highest value combination of all of the streams we've found.
        The flag will be called .use_simple_flow_stream_maximization: bool.
        3.a  When set to true, we simply walk each streams capture lookup table and add a multiple-choice-group knapsack entry for each value=captureLookup[n].econValue weight=(gatherLookup[captureLookup[n].gatherIndex].tileCount + captureLookup[n].tileCount), with each border-pair (each lookup pair) as a 'group' in the MKCP inputs, and then solve MKCP on it for W=turnsToSolve.

            Then the algorithm can attempt to post-optimize the solution with the following methodology:
                Prune the lowest value-per-turn capture options from the results of MKCP, making room to search for further optimization.                 Now, we build a visited set from all the 'still included' gather and capture islands that are included in our current plan.
                We now examine any target islands bordering all current-solution plan sets target islands which are not already visited.
